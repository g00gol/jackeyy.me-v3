# EDHTop16: Revolutionizing MTG Tournament Tracking

EDHTop16 is a comprehensive tournament tracking platform designed specifically for Magic: The Gathering's Commander (EDH) format. What started as a passion project quickly grew into one of the most popular MTG tournament trackers, serving over **100,000+ monthly visitors** and tracking thousands of tournaments worldwide.

## The Problem

The Commander format had a significant gap in tournament data collection and analysis. While other formats had robust tracking systems, Commander players and tournament organizers struggled with:

- **Fragmented data sources** spread across multiple platforms
- **Inconsistent tournament reporting** formats
- **Limited meta-game analysis** tools
- **Poor user experience** on existing platforms

One of the biggest challenges was handling large datasets efficiently:

  const [tournaments, setTournaments] = useState([]);
  const [loading, setLoading] = useState(false);

  const loadMore = useCallback(async () => {
    setLoading(true);

      setLoading(false);
    }
  }, [filters, tournaments.length]);

  return { tournaments, loadMore, loading };
};

### Data Processing Pipeline

Built a robust ETL pipeline to process tournament data:

```python
# FastAPI endpoint for processing tournament submissions
@app.post("/api/tournaments/process")
async def process_tournament_data(tournament_data: TournamentSchema):
    """
    Process and validate tournament submissions
    """
    try:
        # Validate deck lists against Scryfall API
        validated_decks = await validate_deck_lists(tournament_data.decks)

        # Extract meta-game statistics
        meta_stats = analyze_tournament_meta(validated_decks)

        # Store processed data
        result = await db.tournaments.insert_one({
            **tournament_data.dict(),
            "processed_at": datetime.utcnow(),
            "meta_analysis": meta_stats
        })

        return {"status": "success", "tournament_id": str(result.inserted_id)}

    except ValidationError as e:
        raise HTTPException(status_code=422, detail=str(e))
```

### Real-Time Features

Implemented WebSocket connections for live tournament updates:

```javascript
// Real-time tournament updates
const useRealTimeTournaments = () => {
  const [tournaments, setTournaments] = useState([]);

  useEffect(() => {
    const ws = new WebSocket(`${WS_URL}/tournaments`);

    ws.onmessage = (event) => {
      const update = JSON.parse(event.data);
      setTournaments((prev) =>
        prev.map((t) =>
          t.id === update.tournament_id ? { ...t, ...update.data } : t,
        ),
      );
    };

    return () => ws.close();
  }, []);

  return tournaments;
};
```

## Lessons Learned

### Technical Challenges

**Database Design**: Initially struggled with MongoDB schema design for complex tournament structures. Learned to embrace document flexibility while maintaining data consistency.

**API Rate Limiting**: Had to implement sophisticated rate limiting when integrating with external MTG APIs to avoid service disruptions.

**Caching Strategy**: Developed a multi-layered caching approach using Redis and browser caching to handle traffic spikes during major tournaments.

### Product Development

**User Feedback Integration**: Established regular feedback cycles with the community, leading to features like custom meta-game filters and advanced search capabilities.

**Data Quality**: Implemented crowd-sourced validation systems to maintain high data quality while scaling submissions.
